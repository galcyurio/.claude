# 피처 메모리 읽기 (작업 착수 컨텍스트)

피처(Jira 에픽 또는 그 하위 작업)를 **구현·조사·디버깅하기 전에**, 해당 피처의 feature-memory Notion 페이지를 먼저 읽어 컨텍스트를 확보한다. feature-memory는 Jira·Slack·Notion 기획서·API 문서·Figma·GitHub의 결정·미결·영향을 한 페이지에 모아둔 **피처 지식 베이스**다. 이 페이지를 안 읽으면 이미 내려진 결정·미결 사항을 모른 채 작업하게 된다.

> feature-memory를 *쓰는*(갱신) 절차는 `feature-memory` skill이 담당한다. 이 rule은 작업 전 그 결과물을 *읽어* 컨텍스트로 쓰는 연결고리다.

## 자동 트리거

Jira 키(`HDA-\d+` 등)가 등장하는 **구현/조사/디버깅 작업**을 착수할 때:

1. 대상 이슈의 epic_key 또는 부모 에픽 키를 확인한다 (하위 작업이면 Jira `parent`, 에픽이면 자신).
2. feature-memory DB(`collection://314333d2-57ac-4f15-8fa2-12cee61130e1`)에서 그 키로 페이지를 찾는다 — `notion-search(data_source_url=..., query="<Jira project key 또는 epic_key>")` → 후보 `notion-fetch`. `epic_key` property는 쉼표로 여러 키를 묶을 수 있으니(예: `HDA-21839, HDA-21840`) **부분 매칭**으로 찾는다.
3. 페이지가 있으면 본문을 fetch해 아래를 컨텍스트로 흡수한다:
   - `🎯 내가 해야 할 일` · `⚠️ 영향 주는 변경 / 알아야 할 결정` · 미결(⏳) 항목
   - `한눈에 보기`의 핵심 링크와 page property URL(`기획서`·`API 문서`·`Figma`·`Slack`)
   - `📚 Reference`의 API 명세·필터 스펙 등 **서버-클라 계약**
4. 페이지가 없으면 **조용히 skip**한다 (등록 안 된 피처도 많음 — 경고 출력하지 않는다).

## 활용

- 흡수한 **확정 결정**을 작업 방향에 반영한다 (예: "eye 인증은 `filters/` 미포함, 클라 boolean 처리" 같은 결정을 구현에 그대로 적용).
- **미결(⏳) 항목이 현재 작업의 블로커**면 코드부터 짜지 말고 먼저 짚는다.
- API 명세·필터 스펙 등 계약은 메모리의 Reference를 1차 출처로 삼되, 최신 여부가 의심되면 원본(Jira·API 문서)을 재확인한다.
- 작업 중 새 결정·변경이 생기면 `/feature-memory {epic_key}` 갱신을 고려한다 (강제 아님).

## 범위

- 읽기는 가볍게. 페이지 1회 fetch로 충분하면 더 파지 않는다.
- 단순 질의응답, Jira 키 없는 작업, 피처와 무관한 작업에는 적용하지 않는다.
- 메모리 내용은 **작성 시점의 스냅샷**이다 — 파일·필드·플래그를 명시하면 현재도 유효한지 확인 후 사용한다.
