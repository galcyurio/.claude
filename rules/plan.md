# 계획(plan) 파일 작성 규칙

plan/계획 문서를 작성할 때 파일 위치·생성 경로와 무관하게 따르는 공통 규칙이다. `start-jira-plan` skill로 만드는 `.claude/plans/` 아래 파일뿐 아니라, 사용자가 즉석에서 만들어 달라고 요청한 임의 위치의 plan 파일(예: `plan.md`, `docs/plan/*.md`, 대화 중 작성되는 임시 계획 파일)에도 동일하게 적용된다.

## 계획 작성 전: 의도 재진술 및 승인

**계획 본문을 작성하기 전에 요청을 재진술하고 사용자 승인을 받는다.** 초기 프레이밍 오류로 방향이 틀어진 계획을 전부 써놓고 나서 발견하는 최악 케이스를 막기 위한 단계다.

**절차**:

1. 사용자 요청(Jira 이슈 설명, 구두 지시, 직전 대화 맥락)을 바탕으로 핵심 의도를 1~3문장으로 재진술한다.
2. 재진술은 아래를 포함한다:
   - **무엇**: 구체적 결과물
   - **왜**: 배경/동기 (이슈 설명 등에서 확인 가능할 때)
   - **하지 않는 것**: 스코프 밖으로 판단한 항목, 애매했던 부분을 어떻게 해석했는지
3. 재진술 본문을 출력한 **직후 `AskUserQuestion` 도구를 호출하여 승인 여부를 묻는다.** "이 방향이 맞습니까?" 같은 일반 텍스트 질문을 그대로 출력하지 않는다 — 사용자가 옵션 클릭으로 답할 수 있게 한다.
4. 사용자가 교정 응답을 주면 재진술을 수정한 뒤 다시 `AskUserQuestion`으로 확인한다. 승인 전에는 계획 본문을 작성하지 않는다.

**생략 가능한 경우**: 단일 파일 수정, 요청이 자명한 Explicit 작업(예: "X 함수의 오타 수정"), 사용자가 "바로 계획 써줘"라고 명시적으로 지시한 경우.

### `AskUserQuestion` 호출 형식

재진술 본문은 일반 텍스트로 출력한 뒤, 같은 응답에서 `AskUserQuestion`을 호출한다. 권장 파라미터:

- `question`: `재진술한 방향대로 계획을 작성할까요?`
- `header`: `계획 방향` (12자 이내)
- `multiSelect`: `false`
- `options` (2~4개, label/description 필수):
  1. label `재진술대로 진행 (Recommended)` / description `확인된 방향으로 계획 본문 작성을 시작한다`
  2. label `일부 수정` / description `재진술 중 어느 부분을 어떻게 바꿔야 하는지 추가로 알려준다`
  3. label `방향 재정의` / description `재진술 자체를 처음부터 다시 잡는다`

`Other` 옵션은 도구가 자동 제공하므로 직접 추가하지 않는다. 옵션 라벨은 상황에 맞게 다듬되, 첫 번째 항목은 "그대로 진행" 의도로 두고 끝에 `(Recommended)`를 붙인다.

### 재진술 예시

> 제가 이해한 요청은 이렇습니다:
>
> - **무엇**: HDA-1234 로그인 실패 시 재시도 버튼 추가 (`LoginScreen.kt`)
> - **왜**: 5xx 오류 발생 시 사용자가 앱을 재시작해야 하는 불편 해소
> - **하지 않는 것**: 재시도 정책(백오프/횟수 제한)은 이번 스코프 아님. 다른 화면의 에러 처리도 건드리지 않음.

(이어서 같은 응답에서 위 형식으로 `AskUserQuestion`을 호출한다.)

## TL;DR

계획 파일 최상단(제목/메타 바로 아래, 본문 섹션 앞)에 **TL;DR** 섹션을 둔다. 개발자가 이 섹션만 읽고도 작업 맥락을 파악할 수 있어야 한다.

- 이 작업이 **무엇을, 왜 하는지**와 **핵심 변경사항**을 1~3문장으로 요약한다.
- 모호한 표현("개선한다", "정리한다") 대신 구체적 결과를 적는다.

### 템플릿

```markdown
## TL;DR

<이 작업이 무엇을 왜 하는지, 핵심 변경사항을 1~3문장으로 요약한다.>
```

## 작업 항목 작성

- **각 작업 항목은 독립된 커밋 단위로 구성한다.** 여러 변경을 한 번에 묶지 말고, 항목 완료 시점마다 개별 커밋을 생성한다.
- 계획 파일 자체에도 커밋 원칙이 드러나도록 아래 두 장치를 반드시 포함한다:
  1. "작업 항목" 섹션 상단에 커밋 원칙 주석 블록
  2. 각 작업 항목 끝에 `→ 커밋` 마커. 단, 빌드/테스트/타입체크 같은 **검증 항목에는 붙이지 않는다**(검증은 커밋 대상 변경이 아니므로). `→ 커밋`은 표시가 아닌 **필수 게이트**다 — 다음 항목으로 넘어가기 전에 반드시 단독 커밋을 생성한다.
- 각 항목은 하나의 논리적 단위로, 단독으로 검증 가능해야 한다.
- "~를 수정한다" 같은 모호한 표현 대신 "A 파일의 B 함수에서 C를 D로 변경한다" 수준의 구체성을 유지한다.
- 테스트/검증 항목을 반드시 포함한다.

## 작업 항목 템플릿

계획 파일의 "작업 항목" 섹션은 아래 형태를 따른다:

```markdown
## 작업 항목

> **커밋 원칙**: 각 항목은 독립된 커밋 단위다. 항목 완료 시점마다 개별 커밋을 만들고, 여러 항목을 하나의 커밋으로 묶지 않는다. 항목 끝의 `→ 커밋`은 해당 항목 완료 후 커밋을 생성해야 함을 의미한다.

- [ ] 작업 1: 구체적 구현 내용 → 커밋
- [ ] 작업 2: 구체적 구현 내용 → 커밋
- [ ] 작업 3: 테스트 작성/수정 → 커밋
- [ ] 작업 4: 검증 (빌드/테스트/타입체크)
```

## 구현 단계에서의 적용

plan 파일을 따라 구현할 때도 동일 원칙을 적용한다:

- **항목 완료 직후, 다음 항목 시작 전에 반드시** 해당 항목만 스테이징하여 커밋한다.
- 커밋 메시지는 `~/.claude/rules/git-commit-message.md` 규칙을 따른다.
- 여러 항목을 병합해 커밋하지 않는다. 특정 항목이 너무 작다고 판단되면 계획 단계에서 항목을 병합하되, 구현 중 임의로 묶지 않는다.
