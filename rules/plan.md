# 계획(plan) 파일 작성 규칙

plan/계획 문서를 작성할 때 파일 위치·생성 경로와 무관하게 따르는 공통 규칙이다. `start-jira-plan` skill로 만드는 `.claude/plans/` 아래 파일뿐 아니라, 사용자가 즉석에서 만들어 달라고 요청한 임의 위치의 plan 파일(예: `plan.md`, `docs/plan/*.md`, 대화 중 작성되는 임시 계획 파일)에도 동일하게 적용된다.

## 작업 항목 작성

- **각 작업 항목은 독립된 커밋 단위로 구성한다.** 여러 변경을 한 번에 묶지 말고, 항목 완료 시점마다 개별 커밋을 생성한다.
- 계획 파일 자체에도 커밋 원칙이 드러나도록 아래 두 장치를 반드시 포함한다:
  1. "작업 항목" 섹션 상단에 커밋 원칙 주석 블록
  2. 각 작업 항목 끝에 `→ 커밋` 마커. 단, 빌드/테스트/타입체크 같은 **검증 항목에는 붙이지 않는다**(검증은 커밋 대상 변경이 아니므로).
- 각 항목은 하나의 논리적 단위로, 단독으로 검증 가능해야 한다.
- "~를 수정한다" 같은 모호한 표현 대신 "A 파일의 B 함수에서 C를 D로 변경한다" 수준의 구체성을 유지한다.
- 테스트/검증 항목을 반드시 포함한다.

## 작업 항목 템플릿

계획 파일의 "작업 항목" 섹션은 아래 형태를 따른다:

```markdown
## 작업 항목

> **커밋 원칙**: 각 항목은 독립된 커밋 단위다. 항목 완료 시점마다 개별 커밋을 만들고, 여러 항목을 하나의 커밋으로 묶지 않는다. 항목 끝의 `→ 커밋`은 해당 항목 완료 후 커밋을 생성해야 함을 의미한다.

- [ ] 작업 1: 구체적 구현 내용 → 커밋
- [ ] 작업 2: 구체적 구현 내용 → 커밋
- [ ] 작업 3: 테스트 작성/수정 → 커밋
- [ ] 작업 4: 검증 (빌드/테스트/타입체크)
```

## 구현 단계에서의 적용

plan 파일을 따라 구현할 때도 동일 원칙을 적용한다:

- 체크박스 항목을 완료할 때마다 해당 항목만 스테이징하여 커밋한다.
- 커밋 메시지는 `~/.claude/rules/git-commit-message.md` 규칙을 따른다.
- 여러 항목을 병합해 커밋하지 않는다. 특정 항목이 너무 작다고 판단되면 계획 단계에서 항목을 병합하되, 구현 중 임의로 묶지 않는다.
