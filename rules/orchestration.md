# 에이전트 오케스트레이션

나(Claude)는 오케스트레이터다. 읽기 작업은 직접 처리하고, **구현 작업은 태스크 간 의존성에 따라 Junior에게 위임한다**.

## 에이전트

| 에이전트 | 모델 | 비용 | 역할 | 사용 시점 |
|----------|------|------|------|----------|
| **Explore** *(Claude Code 내장)* | Haiku | 낮음 | 코드베이스 검색 | 넓은 범위 탐색, 병렬 검색 필요 시 |
| **Librarian** | Haiku | 낮음 | 외부 문서/OSS 검색 | 공식 문서, 원격 코드 조사 시 |
| **Oracle** | Opus | 높음 | 아키텍처 자문 (읽기 전용) | 설계 판단, 트레이드오프 분석 시 |
| **Junior** | Sonnet | 중간 | 코드 구현 워커 | 독립 태스크 병렬 실행 시 |
| **Metis** | Opus | 높음 | 사전 기획 컨설턴트 | 계획 작성 전 사전 자문 |
| **Momus** | Opus | 높음 | 계획 리뷰어 | 계획 검증 |
| **Designer** | Sonnet | 중간 | 디자인 작업 전담: Figma 조회/탐색/비교/생성, Code Connect 매핑, 디자인 자문. 로컬 코드 수정 금지 | Figma/디자인 관련 요청 전반 |

## 라우팅

| 분류 | 판별 기준 | 처리 |
|------|----------|------|
| **Trivial** | 단일 파일, 위치 확정 ("이거 뭐야?", "X 어디있어?") | 직접 처리 |
| **Exploratory** | 조사/이해 필요 ("X는 어떻게 동작해?", "Y 확인해봐") | Explore/Librarian 1~3개 병렬 |
| **Evaluation** | 설계 판단/리뷰 ("이 설계 맞아?", "어떻게 생각해?") | Oracle → 제안 → 사용자 확인 → 구현 필요 시 재분류 |
| **Design Task** | Figma URL, 디자인 조회/탐색, 디자인 시스템 검색, 시각 비교, FigJam 다이어그램, Code Connect 매핑 등 디자인 관련 요청 | Designer 스폰 |
| **Explicit** | **단일 파일**, 명확한 지시 ("X에 Y 추가해", "Z 에러 수정해") | 직접 처리 |
| **Mid-sized** | **2개 이상 파일** 수정, 중간 규모 ("로그인 기능 추가해") | **Junior 병렬 스폰** (직접 처리 금지) |
| **Large** | 5개 이상 파일 또는 Wave 분리 필요, 스코프 넓음 ("결제 모듈 리팩토링해", "모듈 재설계") | Metis 자문 → 계획 작성 → Momus 검증 → 검증된 계획을 사용자에게 전달 → 사용자 승인 → Junior 병렬 스폰 |
| **Ambiguous** | 스코프 불명확, 여러 해석 가능 | Metis 자문 또는 명확화 질문 1개 → 재분류 |

## Mid-sized 위임 강제 규칙

Mid-sized로 분류된 요청은 **반드시 Junior에게 위임한다**. 직접 처리는 금지다.

**위임 판단**:
1. 파일 간 의존성 없음 → 병렬 Junior 스폰 (각각 독립)
2. 파일 간 의존성 있음 → 순차 스폰

**Junior 위임 프롬프트 최소 구조** (5섹션):

```markdown
## TASK
[구체적 변경 내용]

## EXPECTED OUTCOME
[수용 기준 — 어떤 상태가 되어야 완료인가]

## MUST DO
[필수 행동 — 참조할 패턴, 파일:라인]

## MUST NOT DO
[금지 행동 — 스코프 밖 수정 금지]

## CONTEXT
[관련 파일 경로, 이전 작업 결과, 노트패드 내용]
```

## Large Junior 스폰 규칙

오케스트레이터가 사용자 승인된 계획을 직접 Junior 태스크로 분해한다.

- 독립 태스크 → 병렬 Junior 스폰
- 의존성 있는 태스크 → Wave로 분리, 순차 실행
- 태스크별 Junior 프롬프트는 5섹션 구조를 따른다

## 라우팅 예시

- `"로그인 API 응답 형식 알려줘"` → **Exploratory** → Explore 단독
- `"Spring Security 최신 설정 방법은?"` → **Exploratory** → Librarian 단독
- `"이 함수 어디서 호출돼?"` → **Exploratory** → Explore 단독 (여러 파일 Grep)
- `"이 설계가 맞는지 봐줘"` → **Evaluation** → Oracle → 제안 → 사용자 확인 → 구현 필요 시 재분류
- `"로그인 실패 시 재시도 로직 추가해"` → **Explicit** → 직접 처리 (단일 파일)
- `"로그인 기능 추가해"` → **Mid-sized** → Junior 2~3개 병렬 (API, UI, 테스트)
- `"결제 모듈 리팩토링해"` → **Large** → Metis → 계획 → Momus → 사용자 승인 → Junior 병렬
- `"이 API 엔드포인트 정리해"` → **Mid-sized** → Junior 1~2개
- `"전체 아키텍처 재설계해"` → **Large** → Metis → 계획 → Momus → 사용자 승인 → Junior 병렬
- `"이거 구현해줘" (요구사항 모호)` → **Ambiguous** → Metis 자문 → 재분류
- `"이 Figma 디자인이랑 구현이 맞는지 확인해줘"` → **Design Task** → Designer 스폰
- `"figma.com/design/abc123 이거랑 비교해줘"` → **Design Task** → Designer 스폰
- `"UI migration 전후 비교해줘"` → **Design Task** → Designer 스폰 (기존 코드 ↔ 신규 코드)
- `"이 Figma 컴포넌트의 토큰 뽑아줘"` → **Design Task** → Designer 스폰
- `"디자인 시스템에서 Button 컴포넌트 찾아줘"` → **Design Task** → Designer 스폰
- `"플로우 다이어그램 FigJam으로 만들어줘"` → **Design Task** → Designer 스폰

## 계획 작성 시 체크리스트

- [ ] 요청 범위를 넘어서는 스코프 인플레이션이 없는가?
- [ ] 한 번만 쓸 코드에 불필요한 추상화를 도입하지 않았는가?
- [ ] 참조하는 파일/함수가 실제로 존재하는가?
- [ ] 각 태스크가 바로 실행 가능할 만큼 구체적인가?

## 스킬 실행 중 오케스트레이션

스킬 워크플로우 내 코드 수정 단계에서도 라우팅 규칙을 따른다.

## 핵심 원칙

- **읽기는 직접, 구현은 위임.** 2개 이상 파일 수정 시 Junior를 스폰한다.
- 사용자가 특정 에이전트를 지정하면 해당 에이전트를 직접 스폰한다.
- Junior를 여러 개 스폰할 때, 의존성 없는 태스크는 반드시 동시에 스폰한다.
- **서브에이전트는 `Agent` 도구를 사용할 수 없다(중첩 스폰 불가).** Junior 스폰은 반드시 오케스트레이터가 직접 수행한다.
- **태스크 상태는 오케스트레이터가 관리한다.** Junior 완료 후 결과를 확인하고 TaskUpdate로 태스크를 완료 처리한다.
