## 역할

작업 설명 하나를 받아 **Jira 이슈 생성 → 브랜치 → 구현 → 커밋 → PR 생성**까지 엔드투엔드로 진행한다.
이 스킬은 얇은 오케스트레이터다. 각 단계의 세부 동작은 기존 스킬·규칙에 위임하고, 단계 간 연결과 체크포인트만 담당한다.

| 단계 | 위임 대상 |
|------|----------|
| 이슈 생성 | `create-jira-issue` 스킬 |
| 구현 라우팅 | `~/.claude/rules/orchestration.md` |
| 커밋 | `~/.claude/rules/git.md` + `~/.claude/rules/coding_standard.md` |
| PR 생성 | `create-pr` 스킬 |

## 입력 형식

```
/work <작업 설명>          # 단계별 승인 모드 (기본)
/work auto <작업 설명>     # 완전 자동 모드
/work                      # 설명 생략 → 직전 대화 맥락에서 작업 추론
```

- 첫 토큰이 `auto`이면 완전 자동 모드, 나머지를 작업 설명으로 본다.
- 작업 설명이 비어 있으면 직전 대화 맥락(선택된 코드/파일, 논의 내용)에서 작업을 추론한다. 추론이 모호하면 진행 전 1개 질문으로 확인한다.

## 모드

| 모드 | 체크포인트 | 적합한 작업 |
|------|-----------|-----------|
| **단계별 승인** (기본) | 3개 STOP (① 이슈 후 ② 구현 계획 후 ③ PR 전) | 모든 규모. 사용자가 각 게이트에서 진행/수정 결정 |
| **완전 자동** (`auto`) | STOP 없이 직행 | Explicit / Mid-sized 작업. Large 작업은 `orchestration.md` 크기 게이트가 여전히 적용되어 계획 승인이 강제됨 |

각 STOP은 `AskUserQuestion`으로 진행/수정 여부를 묻는다. 자동 모드는 STOP을 건너뛰되, 단계 실패 시에는 모드와 무관하게 중단한다.

## 실행 방법

### Phase 0. 모드·작업 파싱

1. 입력을 파싱해 모드(`단계별`/`auto`)와 작업 설명을 확정한다.
2. 작업 설명이 없으면 대화 맥락에서 추론한다. 모호하면 1개 질문으로 확인 후 진행.

### Phase 1. 이슈 생성

1. `create-jira-issue` 스킬을 **`single` 모드**로 호출한다.
   - 제목: 작업 설명 기반의 순수 제목 (prefix 없이).
   - 프로젝트 키·App·에픽은 `create-jira-issue`가 현재 브랜치 컨텍스트에서 자동 상속한다.
   - 에픽을 못 찾으면 `create-jira-issue`가 프로젝트 키·App을 사용자에게 묻는다. 자동 모드라도 이 입력은 필요하면 받는다.
2. 생성된 이슈키 `HDA-YYYYY`를 확보한다.
3. **CLAUDE.md 세션 이름 규칙**대로 `/rename` 2줄을 단독 출력한다.
4. **[단계별 모드 STOP ①]** 생성된 이슈(키·제목·설명)를 보여주고, 진행/수정을 `AskUserQuestion`으로 확인한다.

### Phase 2. 브랜치 생성

1. 현재 repo에서 새 브랜치를 만든다.
   - 형식: `feature/HDA-YYYYY-<slug>`
   - `<slug>`: 작업 제목을 짧은 영문 kebab-case로 변환 (한글만 있으면 핵심 의미의 간결한 영문으로).
   - 예: `feature/HDA-20632-login-retry-button`
2. `git checkout -b feature/HDA-YYYYY-<slug>` 실행.
   - 시작 시점 working tree에 커밋되지 않은 변경이 있으면 중단하고 사용자에게 알린다. (작업 섞임 방지)

### Phase 3. 구현

1. 작업을 `orchestration.md` 라우팅 규칙으로 분류한다.
   - 단일 파일 + 명확 → 직접 처리
   - 2+ 파일 (Mid-sized) → Junior 병렬/순차 스폰 (직접 처리 금지)
   - Large → `orchestration.md` Large 절차(계획·검증·승인) 적용
2. **[단계별 모드 STOP ②]** 실제 코드 수정 전, 구현 계획(변경할 파일, 접근, 분해)을 제시하고 `AskUserQuestion`으로 확인한다.
   - 자동 모드: 이 STOP을 건너뛰고 바로 구현. 단 Large로 분류되면 `orchestration.md`가 요구하는 계획 승인은 수행한다.
3. 계획대로 구현한다. 구현 중 범위 이탈 방지를 위해 CLAUDE.md "범위 이탈 방지 절차"(`git status --short` → `git diff` 확인)를 따른다.

### Phase 4. 커밋

1. `coding_standard.md`의 **커밋 단위 분리** 원칙대로 논리 단위마다 커밋한다. 여러 변경을 한 커밋에 몰지 않는다.
2. 커밋 메시지는 `git.md` 포맷 (`HDA-YYYYY 태그: 내용`)을 따른다. 이슈키는 Phase 1에서 만든 키를 사용한다.
3. 구현과 커밋은 보통 단위별로 인터리브된다(구현 한 단위 → 커밋 → 다음 단위).

### Phase 5. PR 생성

1. working tree가 깨끗한지 확인한다 (`create-pr`가 커밋되지 않은 변경이 있으면 종료하므로, Phase 4에서 모두 커밋되어 있어야 한다).
2. **[단계별 모드 STOP ③]** PR 생성 직전, 변경 요약을 보여주고 `AskUserQuestion`으로 최종 확인한다.
3. `create-pr` 스킬을 호출한다. push·base 결정·제목(Jira summary)·본문 작성은 `create-pr`가 처리한다.

## 에러 처리

- **모든 모드 공통**: 어느 phase든 실패하면 그 지점에서 중단하고 무엇이 어디까지 됐는지 보고한다. 이후 phase는 진행하지 않는다.
- 이슈만 생성되고(Phase 1 성공) 구현이 실패하면(Phase 3) → 생성된 이슈와 브랜치를 남겨두고 실패 지점을 보고한다. 이슈를 임의로 삭제하지 않는다.
- Phase 2에서 working tree가 더러우면 → 이슈는 이미 생성됐으므로, 사용자에게 정리 후 재시도를 안내한다.

## 주의사항

- 이 스킬은 **재정의하지 않는다.** 이슈 생성·PR 생성·구현 라우팅·커밋 규칙의 세부는 각 위임 대상이 단일 출처(single source of truth)다. 충돌 시 위임 대상 규칙을 따른다.
- `create-jira-issue`·`create-pr` 호출은 **메인 대화에서 `Skill` 도구로 순차 호출**한다. 스킬은 격리 실행 프레임이 아니라 명령 텍스트 주입이므로, 이 스킬 실행 중에도 다른 스킬을 `Skill` 도구로 호출할 수 있다(서브에이전트 중첩 제한과 무관). 단, **실행 중인 같은 스킬**(`work`)을 재귀 호출하지는 않는다.
- 구현은 **반드시 `orchestration.md` 라우팅**을 거친다. 2+ 파일 수정을 오케스트레이터가 직접 처리하지 않고 Junior에게 위임한다.
- 자동 모드라도 **파괴적·외부 노출 동작**(push, PR 생성)은 그대로 수행한다 — 이 스킬에 진입한 것 자체가 엔드투엔드 진행 동의를 의미한다. 단 단계 실패 시에는 중단한다.
- 단계별/자동 모드 모두 `create-jira-issue`의 App 필수 입력, `create-pr`의 Jira 인증 실패 같은 **위임 대상의 차단 조건**은 그대로 적용된다.
