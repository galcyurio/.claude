## 역할

Jira URL 또는 이슈 키를 입력받아 이슈 제목과 설명을 조회한 뒤, 코드베이스를 탐색하고 해당 내용을 바탕으로 작업 계획을 수립한다.

사용자 입력: $ARGUMENTS

## 입력 형식

다음 중 하나를 제공한다:

- **Jira URL**: `https://xxx.atlassian.net/browse/PROJ-123`
- **이슈 키**: `PROJ-123`

## 실행 방법

### 1. 이슈 키 추출

입력이 URL이면 아래 우선순위로 이슈 키를 추출한다:

1. 경로에 `browse/<키>`가 있으면 그 키 사용
2. 쿼리 파라미터 `selectedIssue=<키>`가 있으면 그 키 사용
3. 위 둘 다 없으면 URL 전체에서 `[A-Z]+-\d+` 패턴의 **첫 번째** 매칭 사용

이슈 키가 직접 입력된 경우(`PROJ-123`) 그대로 사용한다.

지원 URL 예시:
- `https://xxx.atlassian.net/browse/PROJ-123` → 1번 규칙으로 `PROJ-123`
- `https://xxx.atlassian.net/jira/software/projects/PROJ/boards/1?selectedIssue=PROJ-123` → 2번 규칙으로 `PROJ-123`
- `https://xxx.atlassian.net/jira/software/projects/PROJ/issues/PROJ-123` → 3번 규칙으로 `PROJ-123`
- `https://xxx.atlassian.net/browse/PROJ-123?return=TEAM-456` → 1번 규칙으로 `PROJ-123` (쿼리의 `TEAM-456` 무시)

### 2. 이슈 정보 조회

- `mcp__claude_ai_Atlassian__getJiraIssue` MCP 도구로 이슈 정보를 조회한다.
- 필요한 필드: `summary`, `status`, `priority`, `description`, `issuetype`, `comment`, `subtasks`, `parent`, `labels`, `components`
- 호출 실패 시 에러 유형에 따라 대응한다:
  - **인증/권한 오류** → "Atlassian MCP 재인증이 필요합니다" 안내 후 중단
  - **이슈 없음** → "이슈 키를 확인해 주세요: `<입력값>`" 안내 후 중단
  - **그 외 오류** → 에러 메시지를 그대로 노출하고 중단

### 3. 이슈 내용 정리

**제목**: `fields.summary`를 이슈 제목으로 사용한다.

**설명**: `fields.description` 필드를 처리한다:
  - `null`이면 "설명 없음"으로 처리하고 summary만으로 계속 진행한다
  - Atlassian Document Format(ADF) JSON인 경우 `content` 배열을 재귀 탐색하여 markdown으로 변환한다:

| ADF 노드 타입 | 변환 규칙 |
|---|---|
| `heading` | `#` ~ `######` (attrs.level에 따라) |
| `paragraph` | 줄바꿈으로 구분 |
| `text` | 텍스트 그대로. marks가 있으면: `strong` → `**`, `em` → `*`, `code` → `` ` `` |
| `bulletList` | `- ` 접두사 |
| `orderedList` | `1. ` 접두사 (순번 자동 증가) |
| `codeBlock` | ` ``` ` 블록 (attrs.language가 있으면 명시) |
| `table` | markdown 테이블로 변환 |
| `inlineCard` | `[링크](url)` 형태 |
| `mention` | `@displayName` 형태 |
| `mediaGroup` / `mediaSingle` | `[첨부: <filename>]`으로 표기 |
| `hardBreak` | 줄바꿈 |
| 기타/알 수 없는 노드 | 하위 `content`가 있으면 재귀 탐색, 없으면 무시 |

**부가 정보**: 조회된 필드 중 아래 항목이 존재하면 계획 수립 시 참고 컨텍스트로 활용한다:
  - `comment` — 최근 5개까지만 추출하여 핵심 내용을 요약한다
  - `subtasks` — 하위 이슈 목록(키 + 제목 + 상태)을 정리한다
  - `parent` — 상위 이슈가 있으면 키와 제목을 표시한다
  - `labels` / `components` — 관련 모듈/카테고리 힌트로 활용한다

### 4. 코드베이스 탐색

계획 수립 전에 이슈와 관련된 코드를 찾는다:
- 이슈 제목/설명에서 키워드를 추출하여 관련 코드를 찾는다
- `labels`, `components` 정보가 있으면 관련 디렉터리를 우선적으로 살펴본다
- 수정 대상 파일 후보와 관련 모듈 구조를 파악한다
- 기존 코드의 패턴/컨벤션을 확인한다

### 5. Metis 사전 기획 컨설팅

코드베이스 탐색 결과와 이슈 정보를 바탕으로 Metis 에이전트에게 사전 기획 컨설팅을 요청한다.

**Metis 스폰 프롬프트에 포함할 정보**:
- 이슈 제목, 설명, 부가 정보 (코멘트 요약, 하위 이슈 등)
- 코드베이스 탐색에서 발견한 관련 파일, 패턴, 구조
- 수정 대상 파일 후보

**Metis 결과 처리**:
- **사용자 질문이 있는 경우**: Metis가 제시한 질문을 사용자에게 전달하고 답변을 받는다
- **계획 수립자 지시사항**: 계획 파일 작성 시 MUST/MUST NOT 지시사항을 반영한다
- **식별된 리스크**: 계획의 "리스크 및 의존성" 섹션에 포함한다
- **AI 슬롭 경고**: 계획 작성 시 해당 패턴을 회피한다

### 6. 계획 수립 충분성 판단

아래 체크리스트로 바로 계획을 수립할 수 있는지 판단한다:

- [ ] **범위**: 수정 대상 파일/모듈이 특정 가능한가?
- [ ] **접근 방식**: 기술 스택/구현 방향이 결정되어 있는가?
- [ ] **수용 기준**: 완료 조건이 명확한가? (이슈 설명 또는 코멘트에 AC가 있는가?)
- [ ] **의존성**: 선행 작업이나 외부 의존성이 파악되었는가?

→ 모두 충족: Metis 지시사항을 반영하여 계획 수립 진행
→ 하나라도 미충족: Metis 질문과 함께 해당 항목에 대해 사용자에게 추가 질문 후 계획 수립

### 7. 계획 초안 수립

Metis 지시사항(MUST/MUST NOT, 패턴 참조, AI 슬롭 가드레일)을 반영하여 계획 초안을 메모리 상에서 수립한다. **이 단계에서는 파일을 작성하지 않는다.**

계획 초안은 아래 계획 파일 템플릿과 동일한 구조(TL;DR, 배경, 수정 대상, 작업 항목, 검증 기준, 리스크 및 의존성)로 구성한다.

**작업 항목 작성 규칙**:
- **첫 번째 항목은 "작업 브랜치 생성"으로 고정한다.** 브랜치 이름은 `feature/<이슈키>-<slug>` 형식이다 (예: `feature/HDA-12345-login-screen-bug-fix`). `<이슈키>`는 대문자를 그대로 유지하고, `<slug>`는 9번 단계의 slug 생성 규칙을 동일하게 적용한다.
- 각 항목은 하나의 논리적 단위로, 단독으로 검증 가능해야 한다
- **각 작업 항목은 독립된 커밋 단위로 구성한다.** 모든 변경을 한 번에 묶지 말고, 작업 항목 완료 시점마다 커밋한다. 이 원칙이 생성되는 계획 파일에도 드러나야 하므로, 템플릿의 "작업 항목" 섹션 상단 주석과 각 항목의 `→ 커밋` 표기를 반드시 포함한다.
- "~를 수정한다" 같은 모호한 표현 대신 "A 파일의 B 함수에서 C를 D로 변경한다" 수준의 구체성을 유지한다
- 테스트/검증 항목을 반드시 포함한다

### 8. Momus 계획 리뷰

수립한 계획 초안을 Momus 에이전트에게 리뷰 요청한다.

**Momus 스폰 프롬프트에 포함할 정보**:
- 이슈 제목/설명 및 부가 정보 (코멘트 요약, 하위 이슈 등)
- 계획 초안 전체 내용 (TL;DR, 배경, 수정 대상, 작업 항목, 검증 기준, 리스크 및 의존성)
- Metis 지시사항(MUST/MUST NOT, 식별된 리스크)
- "계획 초안을 리뷰하라"는 지시

**판정별 처리**:

- **OKAY**: 계획 확정. 다음 단계(계획 파일 작성)로 진행한다.
- **REJECT**: 블로킹 이슈를 기반으로 계획 초안을 수정한다.
  1. Momus가 제시한 블로킹 이슈(최대 3개)를 확인한다
  2. 계획 초안을 수정하여 블로킹 이슈를 해결한다 (아직 파일로 작성하지 않는다)
  3. 수정된 계획 초안을 Momus에게 재리뷰 요청한다
  4. **최대 2라운드**까지 반복한다. 2라운드 후에도 REJECT이면:
     - 미해결 이슈를 사용자에게 보고하고 판단을 요청한다

### 9. 계획 파일 작성

Momus 리뷰를 통과한 **확정 계획**만 파일로 작성한다.

**파일 경로**: `.claude/plans/<이슈키>-<slug>.md`

**slug 생성 규칙**:
- 이슈 summary를 기반으로 생성한다
- 영문은 소문자로, 한글은 의미 요약 영문 번역으로 변환한다
- 공백/특수문자를 하이픈(`-`)으로 치환한다
- 연속 하이픈을 단일 하이픈으로 정리한다
- 30자 이내로 자른다
- 예: "로그인 화면 버그 수정" → `login-screen-bug-fix`

**계획 파일 충돌 처리**:
- 같은 이슈 키로 시작하는 plan 파일이 이미 존재하면:
  1. 기존 파일 내용을 사용자에게 보여준다
  2. 덮어쓰기 / 새 파일 생성(suffix 추가) / 중단 중 선택을 요청한다

**계획 파일 템플릿**:

```markdown
# <이슈키> <이슈 제목>

> 이슈 타입: <타입> | 상태: <상태> | 우선순위: <우선순위>
> 상위 이슈: <있으면 키 - 제목, 없으면 생략>

## TL;DR

<`~/.claude/rules/plan.md`의 TL;DR 규칙에 따라 1~3문장 요약을 작성한다.>

## 배경

<이슈 설명을 정리한 내용. 코멘트에서 추출한 추가 컨텍스트 포함.>

## 수정 대상

| 파일/모듈 | 변경 내용 |
|---|---|
| `path/to/file.kt` | 변경 설명 |

## 작업 항목

> **커밋 원칙**: 각 항목은 독립된 커밋 단위다. 항목 완료 시점마다 개별 커밋을 만들고, 여러 항목을 하나의 커밋으로 묶지 않는다. 항목 끝의 `→ 커밋`은 해당 항목 완료 후 커밋을 생성해야 함을 의미한다.

- [ ] 작업 브랜치 생성: `feature/<이슈키>-<slug>` 브랜치를 기본 브랜치에서 분기
- [ ] 작업 1: 구체적 구현 내용 → 커밋
- [ ] 작업 2: 구체적 구현 내용 → 커밋
- [ ] 작업 3: 테스트 작성/수정 → 커밋
- [ ] 작업 4: 검증 (빌드/테스트/타입체크)

## 검증 기준

- <수용 기준 1>
- <수용 기준 2>

## 리스크 및 의존성

- <식별된 리스크나 선행 조건>
```

### 10. 결과 표시

- **TL;DR을 가장 먼저 사용자에게 전달한다**: 계획 파일에 작성한 TL;DR 내용을 대화에서 먼저 말해준다
- 이슈 제목, 설명(정리 후), 부가 정보(하위 이슈, 상위 이슈, 코멘트 요약)를 사용자에게 표시한다
- **Momus 리뷰 결과**: 판정(OKAY)과 총평을 표시한다. 수정 라운드가 있었으면 라운드 수를 언급한다.
- 작성된 계획 파일 경로를 안내한다

## 주의사항

- **READ-ONLY**: Jira 상태 전환, 코멘트 추가, 이슈 수정 등의 쓰기 작업은 수행하지 않는다
- **단일 이슈**: 하위 이슈를 자동으로 따라가지 않는다. 단, 목록은 표시한다
